User Guide

Pyana is a sovereign capability runtime designed for AI agents and the applications that orchestrate them. As a user, you interact with pyana through the browser extension wallet, which manages your agent identity, holds authorization tokens, and generates zero-knowledge proofs on your behalf.

What is Pyana?

Pyana lets autonomous agents prove what they are authorized to do without revealing how they obtained that authority. It combines unforgeable capability tokens (like API keys that can only get weaker, never stronger) with zero-knowledge STARK proofs (so verifiers learn only "yes, authorized" or "no, denied").

Agents own their state on their own machine. The federation is a notary service -- used for ordering, double-spend prevention, and discovery -- not as a state host. Between known parties, agents interact directly by exchanging signed proofs. No federation contact, no permission needed.

Key Concepts

Capabilities (Tokens)

A capability token is an unforgeable proof of authority. When you receive one (via delegation from another agent), you can use it to authorize actions. Tokens can only be attenuated (restricted further), never amplified. If you receive "read+write" access to a service, you can create a narrower token with only "read" access to hand off to a sub-agent.

Agent Identity

Your wallet holds an Ed25519 signing key. This is your agent identity. Every action you take (signing turns, generating proofs) is bound to this key. You can derive sub-agent identities from a single mnemonic (BIP39 + BLAKE3).

Federations

A federation is a group of nodes that act as a notary service: providing ordering, nullifier tracking (double-spend prevention), discovery, and root anchoring. Your wallet connects to a federation through a local pyana-node, but most routine interactions between known parties happen directly -- peer-to-peer -- without federation involvement. Think of the federation like DNS: you use it to find people and prove things to strangers, not for every message.

For deeper technical details on how capabilities and proofs work internally, see the Architecture Overview.
