System Design

Pyana is structured as a Rust workspace of 44 crates. The core architecture follows the sovereign cell model: agents maintain their own state and execute transitions locally. The federation verifies proofs and provides ordering -- it does not store or execute on behalf of agents (in sovereign mode). A constraint DSL compiles to 8 proof backends; 21 AIR circuits cover the full authorization and state-transition space.

Data Flow: Authorization Proof

  1. SDK: wallet.authorize(token, request, mode) is called.
  2. Policy: Datalog evaluator runs locally, producing a trace of derived facts.
  3. Bridge: Trace is compiled to circuit witness (BabyBear field elements).
  4. Circuit: AIR constraint system generates a STARK proof (~24 KiB).
  5. SDK: Returns AuthorizationPresentation with proof + selected facts.
  6. Verifier: Checks proof against attested federation root (~2ms).

Data Flow: Turn Execution (Sovereign Mode)

  1. SDK: Build a Turn with a call forest targeting cells.
  2. Agent: Execute the turn locally with the local TurnExecutor.
  3. Circuit: Generate a STARK proof that the state transition is valid.
  4. Submit: Send (turn_hash, new_commitment, proof) to the federation.
  5. Federation: Verify proof, update 32-byte commitment, order if needed.
  6. Wallet: wallet.append_receipt(receipt) extends the proof chain.

Data Flow: Peer-to-Peer (No Federation)

  1. Alice: Executes transition locally, produces STARK proof of validity.
  2. Alice: Sends (old_commitment, new_commitment, proof, signature) to Bob.
  3. Bob: Verifies proof, checks signature, updates his local view of Alice's state.
  4. Federation is never contacted. Storage cost on the network: 0 bytes.

Data Flow: Hosted Mode (Legacy)

  1. SDK: Build a Turn with a call forest targeting cells.
  2. Wallet: Sign the turn with wallet.sign_turn(&turn).
  3. Node API: Submit signed turn to POST /api/turn.
  4. Execution: Federation executes effects atomically (commit or rollback).
  5. Consensus: Federation commits the turn and produces a TurnReceipt.
  6. Wallet: wallet.append_receipt(receipt) extends the proof chain.

Core Insight

Capability attenuation is incrementally verifiable computation.

Each delegation step is a fold over a committed fact set. The fact set tracks which permissions exist, which are narrowed, and the derivation rules that produced them. The STARK proves that this fold chain is valid from root to leaf, without revealing any intermediate step. This is why verification is O(1) regardless of chain length.

For the cryptographic details of how this fold is committed and proven, see Cryptographic Foundations. For the capability semantics (cells, turns, c-lists), see Object-Capability Model.
