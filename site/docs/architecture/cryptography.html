Cryptographic Foundations

Pyana's proof system supports multiple backends. The primary path uses BabyBear-field STARKs with Poseidon2 as the in-circuit hash function, achieving post-quantum security on the verification path (hash-based, no elliptic curve assumptions). Additional backends include Kimchi (Mina/Pasta curves), SP1 (RISC-V zkVM with Groth16 wrapping for EVM), and native Plonky3. A constraint DSL compiles to all backends from a single source.

BabyBear Field

The BabyBear prime field (p = 2^31 - 2^27 + 1 = 2013265921) is optimized for STARK arithmetic. 32-bit limbs mean native CPU word operations with no carries for addition/subtraction in most cases.

Poseidon2

Poseidon2 is the algebraic hash function used inside circuits. It operates natively over BabyBear field elements, making Merkle tree operations and fact commitments cheap inside the AIR (Algebraic Intermediate Representation).

  • hash_fact(predicate, terms) -- Commit a Datalog fact to a single field element
  • hash_2_to_1(left, right) -- Binary Merkle node hash
  • hash_many(inputs) -- Variable-width sponge hash

STARK Proof System

The STARK prover/verifier is based on FRI (Fast Reed-Solomon Interactive Oracle Proofs of Proximity). Key properties:

  • Proof size: ~24 KiB (constant regardless of computation size)
  • Verification time: ~2ms
  • Generation time: 200ms-500ms depending on mode
  • Security: Hash-based (post-quantum), 128-bit security level
  • Transparency: No trusted setup required

AIR Definitions

The pyana-circuit crate defines 21 AIR constraint systems:

  • MultiStepDerivationAir -- Proves a multi-step Datalog derivation is valid
  • MerklePoseidon2StarkAir -- Proves Merkle membership (federation root)
  • FoldAir -- Proves a state transition fold (IVC)
  • NoteSpendingAir -- Proves valid note spending (private transfers)
  • NonRevocationAir -- Proves sorted-Merkle non-membership (non-revocation)
  • PredicateAir -- Proves value predicates (gte, lte, range)
  • AccumulatorAir -- Incremental accumulator updates
  • ArithmeticPredicateAir -- Arithmetic relation proofs
  • BlockTransitionAir -- Valid block/epoch transitions
  • CompoundPredicateAir -- Composed predicate proofs
  • DerivationAir -- Single-step Datalog derivation
  • GarbledAir -- Garbled circuit evaluation
  • NativeSignatureAir -- In-circuit signature verification
  • Poseidon2Air -- Standalone Poseidon2 hashing
  • Plonky3VerifierAir -- Recursive STARK verification in-circuit
  • RelationalPredicateAir -- Multi-column relational predicates
  • SchnorrAir -- Schnorr signature verification
  • SovereignTransitionAir -- Sovereign cell state transitions
  • TemporalAbsenceAir -- Prove non-existence over time range
  • TemporalPredicateAir -- Time-bounded predicate proofs
  • TurnValidityAir -- Full turn validity proof

Proof Backends

  • STARK (primary) -- BabyBear + FRI, ~24 KiB proofs, post-quantum, no trusted setup
  • Kimchi (Mina) -- Pasta curves, IPA commitment, Mina-compatible format. Includes native derivation, fold, IVC, predicates, presentation circuits
  • SP1 -- Succinct's RISC-V zkVM. Proves Datalog execution as ordinary Rust compiled to RISC-V. Groth16 wrapping for EVM verification (~200k gas)
  • Plonky3 -- Native Plonky3 Air impl with recursive composition support
  • Midnight -- ZKIR v3 compilation for Cardano/Midnight proof server (via DSL)

IVC (Incrementally Verifiable Computation)

The IVC system allows receipt chains to be compressed to a constant-size proof. Each new receipt extends the IVC accumulator. The final proof covers the entire state transition history regardless of chain length.

Status: The BabyBear hash-chain IVC is operational with MAX_FOLD_DEPTH=16. Pickles assisted recursion (unbounded depth) is operational. The Mina-equivalent in-circuit IPA verification (Level 1) is in progress -- circuit structure exists but GLV encoding and EndoMul wiring are incomplete. STARK-in-Pickles compression is scaffolded but the full BabyBear Poseidon2 emulation in Kimchi gates is not yet wired.

Signature Schemes

  • Ed25519 -- Agent identity, turn signing, federation consensus messages
  • BLS12-381 -- Threshold signatures for attested roots
  • X25519-ChaCha20Poly1305 -- Sealed capabilities (encrypt under token)
  • HMAC-SHA256 -- Macaroon token chain binding
  • BLAKE3 -- Domain-separated key derivation, fast Merkle trees

EVM Bridge (SP1 Groth16)

The pyana-chain crate wraps pyana STARK proofs in SP1's zkVM to produce Groth16 proofs verifiable on Ethereum/Base at ~200k gas. The architecture:

  1. Pyana STARK proof (large, not EVM-friendly) is the input
  2. SP1 guest program verifies the STARK inside a RISC-V zkVM execution
  3. SP1 produces a compact Groth16 proof
  4. Groth16 proof is verified by SP1 verifier contract on Base/Ethereum

Status: In development. The guest program is being regenerated against the Plonky3 backend. Do not use for real proofs until the guest is updated.

Detailed security analysis and parameter selection rationale are being written.
