Design rationale and formal specification for pyana: a distributed object-capability runtime with zero-knowledge authorization proofs.

Abstract

Summary

We present Pyana, a distributed authorization system that combines object-capability semantics with zero-knowledge proofs to enable agents to prove authorization across organizational boundaries without revealing their delegation chain. The core insight is that capability attenuation — the monotonic narrowing of permissions through delegation — is structurally identical to incrementally verifiable computation, allowing zero-knowledge presentation to emerge naturally from the authorization model rather than being bolted on post hoc.

The system achieves sub-second proof generation (~24 KiB STARK proofs in 300–560ms), sub-millisecond verification, offline operation without blockchain liveness, and post-quantum security for external interfaces through exclusively hash-based verification. Agents carry their own state as receipt chains that compress to constant size via IVC, enabling censorship-resistant exit from any federation.

References

Key references

  • Shapiro & Sela. Cordial Miners: Fast and Efficient Consensus for Every Eventuality. The foundation for pyana's Blocklace federation layer -- DAG-based BFT with 3-round finality and structural equivocation detection.
  • Stingray (arXiv:2501.06531). Bounded counters for budget enforcement without per-operation consensus. Adapted for pyana's multi-silo coordination layer.
  • Mina Protocol. zkApp execution model with call forests and cells-as-accounts. Pyana adapts this architecture for authorization and distributed object semantics.
  • Birgisson et al. (Macaroons). HMAC-chained bearer tokens with cryptographic attenuation. Pyana's token layer supports both Macaroon and Biscuit formats as the pre-ZK authorization substrate.
  • Biscuit (Datalog authorization). Ed25519-signed tokens with embedded Datalog policies. Pyana extends this with STARK proofs over the Datalog derivation trace.
  • Mark Miller (object-capabilities). The E programming language and the principle of least authority. Pyana's execution model (cells, turns, three-party introduction) descends directly from this lineage.
