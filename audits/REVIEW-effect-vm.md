# Pyana Effect VM — Protocol Review

**Verdict:** INCOHERENT (resolvable; the abstract spec is *near* coherent but the runtime/AIR seam destroys the headline guarantees).

---

## Executive Summary

The Effect VM advertises itself as the o1vm-equivalent for pyana: a single STARK that proves an arbitrary sequence of 24 effects took the cell from `old_commitment` to `new_commitment`, with a conservation tag (`net_delta`), an effect-sequence commitment (`effects_hash`), and a slot for `MAX_CUSTOM_EFFECTS = 4` external program proofs. The trace is `selector[24] | state_before[14] | params[8] | state_after[14] | aux[11]`, one row per effect, padded with `NoOp`. Constraints are gated by selector. Continuity, commitment-tree binding, and a single boolean sign bit on `net_delta` are the only structural invariants; balance limb ranges and underflow are explicitly outsourced to the executor. That is a defensible STARK design at the AIR layer in isolation.

The protocol fractures at three seams.

**Seam 1 — Two `Effect` enums.** There is the AIR's `circuit::effect_vm::Effect` (24 variants, BabyBear-typed, no `cell` identity) and the runtime's `turn::action::Effect` (~40 variants, byte-typed, addressing cells by `CellId`). The runtime's variants do not bijection with the AIR's variants. The bridge `convert_turn_effects_to_vm` (turn/src/executor.rs:1254) is a *lossy* projection: ~half the runtime effects become `Effect::NoOp` (line 1429–1432), several get default/placeholder parameters (e.g. `EnqueueMessage.queue_len = 0`, `DequeueMessage.expected_message_hash = hash(queue_id)`, `AtomicQueueTx.combined_old_root = combined_new_root`), and only effects targeting the proven cell are emitted. Because the executor commits the proof against PIs derived from this projection (turn/src/executor.rs:966–998), the proof attests a *fictional* effect sequence — not the actual turn. This is the headline P0.

**Seam 2 — Two `CellState` shapes.** The runtime cell (cell/src/state.rs) has 8 fields of 32 bytes plus visibility, commitments, balance, nonce, `proved_state`, `delegation_epoch`, sealed-field state living in `Permissions`, capabilities living in a separate `CapabilityTable`, etc. The AIR cell (circuit/src/effect_vm.rs:621) has 8 *BabyBear* field slots, a u64 balance split into 30+34-bit limbs, `capability_root` as one hash, `sealed_field_mask` and `mode_flag` co-packed into a `reserved` slot, and the rest of the commitment tree zero-padded. Reductions from runtime field elements to AIR field elements (turn/src/executor.rs:1271–1273) take the *first 4 bytes mod p* — every 32-byte field element collapses to ~30 bits. So `state_commitment` in the AIR sense is collision-trivial relative to the runtime state it claims to bind, and the on-disk sovereign commitment is stored as a `[u8;32]` whose last 28 bytes are required zero (turn/src/executor.rs:1225, `commitment_to_babybear`). It is a 31-bit commitment masquerading as a 256-bit commitment.

**Seam 3 — Aspirational naming.** "Atomic" `AtomicQueueTx` is a single AIR row whose witness collapses `combined_old_root == combined_new_root` (turn/src/executor.rs:1399). "Sealed" fields are unverified: the AIR's `Seal` and `Unseal` effects only check that `field_idx ∈ {0..7}`; they leave `reserved` unchanged, and `SetField` does not consult the sealed mask at all. "ValidateHandoff" proves `aux[0] == hash_2_to_1(cert_hash, approved_root)` — i.e. *a hash equals itself*, not Merkle membership. "PipelineStep"'s `pipeline_id` parameter is bound to nothing (it is loaded as `_pipeline_id_val` and discarded). "EnqueueMessage" updates `field[4]` as a hash chain whose hash semantics encode FIFO incorrectly (the same hash chain is used for both enqueue and dequeue, so dequeue can only "consume" the most recent enqueue if you read the constraints literally — this matches a stack, not a queue).

The single strongest design choice is the o1vm-like *trace-row-per-effect* shape with a unified selector. It cleanly bounds the AIR width and degree, supports padding, and makes the public-input layout uniform. The single weakest is that *every* effect that is not balance-arithmetic-or-hash-chain-typed reduces to a no-op constraint plus a name in `effects_hash`. Seal/Unseal/MakeSovereign/CreateCellFromFactory/Custom all change the trace selector but not the state. So `effects_hash` is the only thing the proof actually attests for the vast majority of effect variants. The system is structurally an *effects-hash binder* with a *balance-and-cap-root state machine*, not a 24-instruction VM.

The design is recoverable: pick one `Effect` enum (or formalize the projection); widen the state-commitment domain to a multi-element hash; verify what each effect's name claims to verify (or rename them honestly). The pieces are there, the seams are not.

## Findings

### P0 — Design contradictions

1. **Runtime→AIR lossy projection silently truncates effect semantics.**
   `turn/src/executor.rs:1429-1432` maps any unrecognized runtime `Effect` to `VmEffect::NoOp`. The committed proof's `effects_hash` (turn/src/executor.rs:967) is computed over the *projected* sequence, not the runtime call forest. So a turn containing e.g. `ReleaseEscrow`, `BridgeMint`, `ExerciseViaCapability`, `Introduce`, `PipelinedSend`, `CreateCommittedEscrow`, `RefundEscrow`, `BridgeLock`, `RefreshDelegation`, `RevokeDelegation`, `BridgeFinalize`, `BridgeCancel`, `Seal`/`Unseal`, `SetPermissions`, `SetVerificationKey`, `CreateCell`, `EmitEvent`, `CreateSealPair`, `IncrementNonce`, `RevokeCapability`, `MakeSovereign`, `CreateCellFromFactory`, `SpawnWithDelegation`, `CreateObligation`, `FulfillObligation`, `SlashObligation`, `QueuePipelineStep` is proven as a sequence of NoOps and a few transfers. Field: a sovereign cell whose program proof verifies a circuit-side `effects_hash` of *transfers only* will accept turns containing arbitrary auxiliary effects with no AIR coverage.
2. **Sovereign state commitment is ~31 bits, not 256.**
   `commitment_to_babybear` (turn/src/executor.rs:1225) reads the first 4 bytes of the stored 32-byte commitment as a `u32 mod p`. The state_commitment column is a *single* BabyBear element. Birthday on a 31-bit hash is ~2^15.5 ≈ 50K trials. Any "binding" claim about old/new commitments has cryptographic strength comparable to a 4-digit PIN.
3. **`Seal` / `Unseal` do not actually seal.** The AIR's Seal/Unseal constraint blocks (effect_vm.rs:1451-1513) constrain all state to be unchanged. `sealed_field_mask` lives in the high bits of `reserved`, but no Seal constraint updates `reserved` and no `SetField` constraint reads it. The state-commitment tree (effect_vm.rs:2266-2271) does hash `reserved` (because `reserved` is included via the `to_trace_cols` packing, wait — actually, **inspect**: the tree hashes `bal_lo, bal_hi, nonce, field[0..7], cap_root, ZERO`. **`reserved` is not in the commitment tree at all.** So seal state has no cryptographic identity. **Compounding P0.**
4. **`ValidateHandoff` proves a tautology.** `aux[0] = hash_2_to_1(cert_hash, approved_root)`: both inputs are public-input parameters, and `aux[0]` is a prover-supplied witness. The constraint `aux[0] - hash(cert_hash, approved_root) == 0` is satisfied by computing the hash. No Merkle path. The cap_root update *does* bind `(recipient_pk, cert_hash)` into the cap-root chain, but anyone holding any cert + any "approved root" can produce a valid handoff. (Mirrors a known anti-aspirational-naming finding in audits/AUDIT-wallet.md style.)
5. **`EnlivenRef` proves no swiss table membership.** Same shape as #4: `aux[0] = hash(swiss, hash(cell_id, perms))` is satisfied tautologically. The "table" is not committed anywhere.
6. **`DequeueMessage` uses the enqueue hash-chain formula.** `new_root = hash(old_root, expected_msg)` — appending, not consuming. A real FIFO needs either: (a) a witness path through a Merkle accumulator with head/tail pointers, or (b) hash-chain reversal (i.e. proving `old_root = hash(new_root, expected_msg)`). The current constraint cannot distinguish dequeue from enqueue. Field: a malicious sovereign can "dequeue" any message they like.
7. **`AtomicQueueTx`'s "atomicity" is a name.** A single AIR row with `combined_old_root` → `combined_new_root` carries no information about how many ops, which queues, or in what order. The bridging code (executor.rs:1396-1402) collapses `combined_old_root == combined_new_root`, which is the maximally-empty case. There is no in-AIR enforcement that the sub-op set actually composes to the declared transition; `tx_hash` only binds the *declared* ops to the prover's witness — not to any queue's actual state.
8. **`PipelineStep.pipeline_id` is unbound.** `effect_vm.rs:2068` reads it into `_pipeline_id_val` and never uses it. The "proves authorization" comment (effect_vm.rs:2099-2102) describes intent, not enforcement.
9. **`ResizeQueue` BabyBear-wraps on shrink.** `delta = new_capacity - old_capacity` is field subtraction; for `new < old`, delta becomes `p - (old-new)`. The constraint `new_bal_lo - old_bal_lo + delta*cost = 0` then *debits* the wrapped value. The executor's mitigation does not exist for the standalone-AIR ("trustless") path.
10. **`custom_count` in PI is not constrained to match the selector trace.** `pi::CUSTOM_EFFECT_COUNT` (PI[6]) is read by witness generation only. There is no AIR constraint that the number of `s_custom == 1` rows equals `PI[6]`. A prover can declare 0 custom proofs while running 3 custom rows; the executor's external verification loop (executor.rs:1083) iterates over `PI`-declared entries only.

### P1 — Design gaps

11. **`SlashObligation` credits the proven cell, not the beneficiary.** The denotational doc (line 438-447) says "transfer locked stake to beneficiary," but the AIR row exists in the obligor's trace and increments *that* cell's balance. Routing is executor-side.
12. **`MakeSovereign` is not idempotent and not boolean-valued.** Doubling adds 256 twice; nothing constrains `mode_flag ∈ {0, 1}` or that the transition is once-only.
13. **`ExportSturdyRef.counter` parameter is not bound to `field[7]` pre-state.** The constraint only checks `new_f7 = old_f7 + 1`, while the swiss-number derivation uses `param.counter`. Prover can supply any `counter` value; collision potential.
14. **`DropRef.holder_federation` is bound to nothing.** Effectively decorative.
15. **`AllocateQueue` cost wraps without range checks.** `capacity * cost_per_slot` in BabyBear.
16. **The on-disk commitment encoding is asymmetric.** Stored: 32 bytes; used: 4 bytes. `babybear_to_commitment` zero-extends, so a verifier comparing 32 bytes will accept only `[low4][zero28]`. This is fine bytewise but pretends to be a 256-bit commitment in storage.
17. **`effects_hash` uses two 31-bit elements (`hi = hash_2_to_1(h, 0xEFFEC7)`).** That second hash is deterministic from the first — adds zero entropy. The "wider coverage" comment (effect_vm.rs:961) is wishful; `(lo, hi)` is `(h, f(h))`. Effective binding: ~31 bits.
18. **`Transfer` only writes to `bal_lo`; `bal_hi` is constrained equal across the row.** For `balance > 2^30`, you cannot represent a Transfer in this VM. The runtime supports `u64` balances; the AIR can only prove on the bottom 30 bits.
19. **`net_delta` is a `u32` (BabyBear-sized).** Signed totals exceeding 2^31 are unrepresentable. For a multi-Transfer turn, net delta in PI may not equal the actual net delta of the runtime turn.
20. **Custom-effect's `proof_commitment` is hashed via `blake3(proof_bytes)[..16]` and packed into 4 BabyBear elements.** That's 124 bits of binding to a *Blake3-truncated* commitment, but the AIR side computes its own `effects_hash` over the prover-declared 4 BabyBears — so a malicious prover with control over `proof_commitment` (i.e. ability to choose `proof_bytes` with a target prefix) can grind 124 bits. Cost is high but lower than 256.
21. **`PI[CUSTOM_EFFECT_COUNT]` constant is 6, while `CUSTOM_PROOFS_BASE` is 7.** Naming overlap with `BASE_COUNT = 7` and `MAX_CUSTOM_EFFECTS = 4` — the PI layout depends on `custom_count` and there is no AIR constraint that bounds `custom_count ≤ MAX_CUSTOM_EFFECTS`.

### P2 — Rough edges

22. **No identity effect at the category-theory level.** `NoOp` exists, but because `Custom`, `MakeSovereign`, `CreateCellFromFactory`, `Seal`, `Unseal`, and a few others *also* keep state unchanged, the category has many parallel identity-shaped morphisms with different selectors. The category lacks a unique identity.
23. **Queue effects mix two `field[]` slots non-uniformly.** AllocateQueue/Enqueue/Dequeue/AtomicTx/Pipeline all use `field[4]` as "queue root", while ResizeQueue uses `field[5]` for capacity. Programmable queues' program-VK lives in `param[4]` rather than state. There is no formal queue-cell type.
24. **CapTP effects scatter state across `field[5]` (refcount), `field[6]` (use count), `field[7]` (export counter).** No structure binds these to a "CapTP-cell" type.
25. **`compute_effects_hash` is order-sensitive but the order is whatever the executor produced.** Order matters for the AIR row sequence but the runtime's projection (`convert_turn_effects_to_vm`) walks the call forest DFS over a *single cell*. If two effects target different cells, only the proven cell sees its slice — the cross-cell order between *cells* is lost.
26. **The `effect_interp.rs` interpreter (1683 lines) is a parallel semantics implementation.** It exists alongside witness gen but is not what the executor uses for runtime application. Three different "what does this effect do" sources of truth in the tree (runtime executor, AIR witness, effect_interp). Drift risk.

### P3 — Design opinion

27. **State commitment domain.** A single BabyBear element is too narrow for the named domain. A 4-element wide-hash (as used in non-revocation and presentation proofs) would close most P0 cryptographic-binding findings.
28. **Effect categories.** Group effects into orthogonal sub-VMs by *what they touch*: balance-arithmetic (Transfer/Note*/Obligation/EnqueueDeposit), cap-list-hash-chain (GrantCap/Handoff/Slash), field-mutation (SetField/Seal/Unseal), queue-state (Allocate/Enqueue/Dequeue/Resize/Atomic/Pipeline), CapTP (Export/Enliven/Drop), structural (Custom/MakeSovereign/Factory/NoOp). Each sub-VM can have its own AIR or its own column band. This matches what is *already informally true* about which fields each effect class touches.
29. **Either the runtime `Effect` enum or the AIR `Effect` enum should be derived from the other.** Right now both are hand-written and the mapping is a 200-line `match` in the executor.

## Composition Matrix (effect-class buckets)

Buckets:
- **Bal** = balance-arithmetic effects: Transfer, NoteSpend, NoteCreate, CreateObligation, FulfillObligation, SlashObligation, AllocateQueue, EnqueueMessage, DequeueMessage, ResizeQueue, AtomicQueueTx.
- **Cap** = cap-root-mutating effects: GrantCapability, SlashObligation, ValidateHandoff.
- **Field** = field-mutating: SetField, Seal, Unseal, ExportSturdyRef (field[7]), EnlivenRef (field[6]), DropRef (field[5]), AllocateQueue/Enqueue/Dequeue/Resize/Atomic/Pipeline (field[4]/[5]).
- **Id** = state-identity effects: NoOp, Custom, CreateCellFromFactory.
- **Mode** = MakeSovereign.

Commutativity in the AIR (i.e. swapping two adjacent rows yields the same final state):

|         | Bal | Cap | Field | Id | Mode |
|---------|-----|-----|-------|----|----|
| **Bal**  | ✗ when same lo limb crosses 2^30 boundary; **wrap-unsafe** | ✓ disjoint columns | ✗ if `Field` includes balance-affecting queue effects | ✓ | ✓ |
| **Cap**  | ✓ | ✗ hash chain order-dependent | ✓ | ✓ | ✓ |
| **Field**| ✗ shared columns (queue, counters) | ✓ | ✗ on same idx | ✓ | ✓ |
| **Id**   | ✓ | ✓ | ✓ | ✓ | ✓ |
| **Mode** | ✓ | ✓ | ✓ | ✓ | ✗ (idempotency violated — see P1.12) |

Key observations:
- Cap-chain effects (Grant, Handoff, Slash) are *all* writing to the same `cap_root` via hash chains; reordering changes the chain. This is correct but the AIR does not document the canonical order; the witness generator chooses.
- Field-class effects share `field[4]` (queue root) across AllocateQueue/Enqueue/Dequeue/Atomic/Pipeline. Reordering changes the hash chain. There is no AIR constraint that effects on different queues stay segregated — `field[4]` is a single slot for all queues this cell has.
- A cell with more than one queue is structurally unrepresentable in the AIR's state. The runtime allocates queues as separate cells; this is OK in practice but not by AIR-level invariant.

## Open Questions for the Designer

1. **Is the runtime `Effect` enum supposed to be a superset that *projects* onto the AIR `Effect` enum, or are they intended to be isomorphic?** If projection: where is the spec that says which runtime effects produce no AIR row, and what proof object covers those effects? If isomorphism: there is significant unfinished work in the bridge (executor.rs:1276).
2. **What does `state_commitment` cryptographically bind?** Is the design intent a single BabyBear element (~31 bits, which seems too narrow for "binding") or a wide hash (which is what the docs imply and what other proofs use)? If the former, can we live with 31-bit binding? If the latter, the state-commitment domain needs to be 4 elements like the rest of the system's wide hashes.
3. **Are `Seal`/`Unseal` intended to be enforced by the Effect VM, or are they purely runtime concepts that the AIR records only by name?** If the former: the `reserved` slot must be in the commitment tree and SetField must read the mask. If the latter: rename to `NoteSeal`/`NoteUnseal` and document them as `effects_hash`-only annotations.
4. **For `ValidateHandoff` and `EnlivenRef`: where does the "approved set" or "swiss table" actually live?** Both should bind to a Merkle root that is either part of the public input or part of the cap_root. Without that, these effects are decorative.
5. **`DequeueMessage`'s constraint is the same shape as `EnqueueMessage`'s. Is the intent a stack (LIFO) or a queue (FIFO)?** If FIFO, we need a head-pointer or accumulator design. If LIFO, the names are wrong.
6. **What should the AIR-level guarantee of `AtomicQueueTx` be?** Option A: a separate atomic-tx sub-AIR with one row per op and a "combined" boundary constraint. Option B: drop the AIR-level claim and document it as an executor-only effect.
7. **Should `Custom` rows count toward `PI[CUSTOM_EFFECT_COUNT]` via an in-AIR sum constraint** (`sum of s_custom across rows == PI[CUSTOM_EFFECT_COUNT]`), so that the verifier's external-proof loop is guaranteed to cover *every* Custom row?
